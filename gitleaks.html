Gitleaks - Detecção de Credenciais

O Gitleaks é uma ferramenta de detecção de secrets (credenciais, tokens, chaves de API) comprometidas no código-fonte e histórico do Git. A execução deve ocorrer tanto localmente quanto no pipeline CI/CD.

1. Visão Geral

TIPO

Secrets Detection

LINGUAGEM

Agnóstico (All)

EXECUÇÃO

Local + Pipeline

2. Instalação (Linux)

O desenvolvedor deve garantir que o binário está acessível globalmente no sistema para auditorias rápidas e eficientes.

INSTALAÇÃO VIA BINÁRIO

Bash 
# Baixar a release estável do Gitleaks
wget https://github.com/gitleaks/gitleaks/releases/download/v8.18.4/gitleaks_8.18.4_linux_x64.tar.gz

# Extrair os arquivos do pacote
tar -xf gitleaks_8.18.4_linux_x64.tar.gz

# Mover o binário para o diretório de binários do sistema
sudo mv gitleaks /usr/local/bin/

# Verificar se a instalação foi bem-sucedida
gitleaks version

3. Fluxo de Trabalho do Desenvolvedor

Seguir este passo a passo garante que nenhum segredo chegue ao repositório remoto.

Auditoria Local

A ferramenta deve ser executada obrigatoriamente dentro da pasta raiz do projeto antes de cada push.

VERIFICAÇÃO ANTES DO PUSH

Terminal 
# 1. Navegue até a pasta do repositório
cd /caminho/do/seu/projeto/

# 2. Execute a detecção no histórico local com log detalhado
gitleaks detect -v

4. Entendendo o Report

O Gitleaks exibe um log detalhado. Use os botões abaixo para entender cada campo do relatório gerado:

TERMO: SECRET
PcK.iE1W5Rteste90bibzypRypzUuFE

Representa a credencial exposta encontrada. No exemplo, trata-se de um token do HashiCorp Vault que nunca deve estar no código.

TERMO: FILE & LINE
File: helm/templates/secret-vault.yaml
Line: 7

Indica o caminho do arquivo e a linha exata onde o segredo foi detectado. Facilita a localização para remoção imediata.

TERMO: COMMIT & AUTHOR
Author: Explanador De Senhas
Email: explanador.senhas@ufla.br

Mostra quem realizou o commit com o segredo. Isso ajuda a identificar o responsável para orientações de segurança e rotação de chaves.

TERMO: FINGERPRINT
e758b597a9d8f41f80a349fd4fc809ffcf36c4e5:helm/templates/secret-vault.yaml:generic-api-key:7

A impressão digital única do vazamento. É este valor que você usa no arquivo .gitleaksignore se precisar ignorar um falso positivo.

5. O que fazer se achar uma credencial?

Adotamos uma abordagem de tolerância zero. Ao detectar um segredo, ele deve ser expulso e tratado imediatamente.

1º PASSO: ELIMINAÇÃO

Delete a linha que contém o segredo exposto no código fonte.

2º PASSO: REVOGAÇÃO

Revogue a chave antiga no serviço de origem e gere uma nova credencial.

ARMAZENAMENTO CORRETO

FORMA ERRADA

Inserir credenciais diretamente no código (Hardcoded), comentários ou arquivos .env sem estarem no .gitignore.

FORMA CERTA

Variáveis de ambiente (Server-side), Secret Managers (HashiCorp Vault) ou arquivos de configuração ignorados pelo Git.

EXPURGO HISTÓRICO (LIMPEZA DO GIT)

A memória do Git não esquece!

Apenas apagar a senha e fazer um novo commit não resolve. A credencial continuará no histórico. Utilize uma das alternativas abaixo:

ATENÇÃO: BRANCHES COMPARTILHADAS

Nunca execute a reescrita de histórico (BFG ou Filter-Repo) em branches públicas como main ou develop sem alinhar com todo o time. Isso causará conflitos irreversíveis nos repositórios locais dos outros desenvolvedores.

OPÇÃO A: BFG REPO-CLEANER 
# 1. Crie um arquivo com o segredo a remover
echo "SUA_CHAVE_AQUI" > segredos_para_remover.txt

# 2. Substitua o segredo por "*** REMOVED ***" no histórico
bfg --replace-text segredos_para_remover.txt

# 3. Limpe referências e force o push
git reflog expire --expire=now --all && git gc --prune=now --aggressive
git push --force
OPÇÃO B: GIT FILTER-REPO 
# Substitui o segredo em todo o histórico de uma vez
git filter-repo --replace-text <(echo "CHAVE_ANTIGA==>***REMOVED***")

# Sincroniza com o servidor remoto
git push origin --force --all

6. Falso Positivo e Waivers

CONDIÇÃO DE ACEITE
  • O processo de Waiver é exclusivo para Falsos Positivos.
  • É ESTRITAMENTE PROIBIDO registrar Waiver para credenciais reais. Se é real, deve ser rotacionada.

Existem duas formas principais de silenciar um falso positivo no GEDAI:

1. VIA .GITLEAKSIGNORE (FINGERPRINT)

Utilize quando o vazamento é específico de uma linha ou commit.

Passo 1º: Obter o Fingerprint
Terminal
gitleaks detect -v -r relatorio.json -f json
Passo 2º: Adicionar ao ignore
Terminal
echo "FINGERPRINT_COPIADO" >> .gitleaksignore

2. VIA GITLEAKS.TOML (ALLOWLIST)

Utilize quando precisa ignorar arquivos inteiros ou pastas.

gitleaks.toml
[allowlist]
paths = ['''doc/manual.pdf''', '''tests/mock_data.csv''']
3. Registro de Justificativa Commit (Obrigatório)

Ao commitar o arquivo de ignore, utilize o modelo padrão para rastreabilidade:

Template de Commit
Waiver de Segurança: [Gitleaks-FP]
Justificativa: String detectada é um hash MD5 de integridade, não uma credencial.
Status: Validado como Falso Positivo.

4. Organização do Waiver no OpenProject

Crie um ticket para fins de auditoria seguindo rigorosamente o padrão abaixo:

  • ASSUNTO [WAIVER] - [Gitleaks] - [Nome do Arquivo]
  • ETIQUETAS Segurança Waiver
  • DESCRIÇÃO Deve conter a Justificativa Técnica detalhada e o Link do Commit onde o waiver foi aplicado.
