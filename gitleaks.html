Gitleaks - Detecção de Credenciais

O Gitleaks monitora e previne o vazamento de segredos no histórico do Git. Este manual foca na geração de evidências e no processo de Waiver.

1. Visão Geral

TIPO

Secrets Detection

LINGUAGEM

Agnóstico (All)

EXECUÇÃO

Local + Pipeline

2. Instalação (Linux)

Bash 
wget https://github.com/gitleaks/gitleaks/releases/download/v8.18.4/gitleaks_8.18.4_linux_x64.tar.gz
tar -xf gitleaks_8.18.4_linux_x64.tar.gz
sudo mv gitleaks /usr/local/bin/
gitleaks version

3. Fluxo de Trabalho

Geração de Evidências

Para processos de Waiver, é obrigatório gerar o relatório em formato JSON.

Gerar Relatório JSON ou CSV 
# Salva o Relatório de falhas em um arquivo json
gitleaks detect -v --report-path gitleaks-report.json -f json
# Salva o Relatório de falhas em um arquivo Csv
gitleaks detect -v --report-path gitleaks-report.csv -f csv

4. Campos do Relatório

Secret

A credencial real detectada.

File & Line

Responsável pelo commit.

Fingerprint

ID único para Waiver.

e758b597a9d8...

5. O que fazer se achar uma credencial?

🗑️ 1. ELIMINAÇÃO

Apague a linha do código e faça o expurgo do histórico Git.

🔄 2. REVOGAÇÃO

Invalide a chave no serviço de origem e gere uma nova credencial.

6. Falso Positivo e Waivers

AVISO

Waivers são exclusivos para strings que parecem senhas mas não são. Segredos reais nunca ganham Waiver.

PADRÃO DE COMMIT (FALSO POSITIVO)

Ao adicionar um Fingerprint no .gitleaksignore, utilize obrigatoriamente este padrão de mensagem:

Commit Template 
Waiver de Segurança: [Gitleaks-FP]
Justificativa: A string detectada é um exemplo de documentação/hash de integridade.
Status: Validado como Falso Positivo.

REGISTRO NO OPENPROJECT (OBRIGATÓRIO)

Atenção: É mandatório anexar o arquivo gitleaks-report.json gerado no ticket.

  • ASSUNTO: [WAIVER] [Gitleaks] - [Nome do Arquivo]
  • ANEXO: gitleaks-report.json
  • DESCRIÇÃO: Justificativa técnica + Link do Commit de Waiver.

7. Ignorando Credenciais Específicas (.gitleaksignore)

O arquivo .gitleaksignore funciona como uma "lista de permissões" para ocorrências específicas (falsos positivos) que já foram validadas.

.gitleaksignore 
# Ignora chaves de teste específicas em arquivos de regras
418edf165dbb63d6f46993ae8f8818ffd87ea582:cmd/generate/config/rules/jwt.go:jwt:17
525d9792b1e3670b4630b8fcc385ca22e8544f9b:detect/detect_test.go:sidekiq-secret:120
717cf1b10be1625875199eca8cdf48883348985f:README.md:aws-access-token:23

8. Ignorando Arquivos e Pastas (.gitleaks.toml)

Para ignorar diretórios inteiros (como testdata ou documentações) e evitar que o Gitleaks perca tempo os analisando, crie um arquivo .gitleaks.toml na raiz estendendo as regras originais.

.gitleaks.toml 
# This is NOT the default gitleaks config. The default gitleaks config is located
# here: https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml
# This gitleaks config extends the default config and ignores secrets located in the `cmd/generate/config/rules`

title = "Gitleaks development"
[extend]
useDefault = true

[allowlist]
paths = [
  '''(^|/)cmd/generate/config/rules''',
  '''.*test\.go''',
  '''testdata'''
]
Rodando com Custom Config

Para aplicar as regras do seu .toml, adicione a flag --config no comando de execução.

Execução Customizada 
# Detecta falhas ignorando os arquivos listados no .gitleaks.toml
gitleaks detect -v --config .gitleaks.toml
