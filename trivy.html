Trivy - Software Composition Analysis

O Trivy é a nossa principal ferramenta de visibilidade para a Supply Chain, identificando vulnerabilidades em dependências, imagens de container e infraestrutura como código (IaC).

TRIVY: O NOSSO DETETIVE

O Trivy atua como um investigador implacável: ele vasculha cada camada do projeto para encontrar o crime (vulnerabilidade). No entanto, ele apenas reporta os fatos. A sentença final — se vamos atualizar, ignorar ou mitigar — é ditada estritamente pela nossa Triagem de Bibliotecas.

1. Avaliação dos Resultados após rodar o Trivy

Ao executar o scanner, o Trivy apresenta uma tabela de resultados parecida com este exemplo abaixo:

backend/package-lock.json (npm)
Total: 8 (UNKNOWN: 0, LOW: 0, MEDIUM: 4, HIGH: 4, CRITICAL: 0)
Library Vulnerability Severity Fixed Version
@remix-run/router CVE-2026-22029 HIGH 1.23.2
body-parser CVE-2025-13466 MEDIUM 2.2.1

Por que a biblioteca não está no meu arquivo?

É comum pesquisar por @remix-run/router no seu pom.xml ou package.json e não encontrar resultados. Isso ocorre porque você declara apenas as bibliotecas "pai" (diretas).

O Trivy, porém, reconstrói o grafo completo para identificar falhas em dependências transitivas. Ou seja, ele encontra vulnerabilidades nos "filhos" e "netos" das bibliotecas que você instalou.

Entenda a lógica: Quando o relatório aponta o seu arquivo de manifesto como o Target, ele não está dizendo que o erro está escrito ali textualmente, mas sim que aquele arquivo é a origem da cadeia de importação.

Para o desenvolvedor, o desafio é identificar qual "pai" trouxe esse "filho" vulnerável. Corrigir a versão da biblioteca pai no seu arquivo geralmente resolve as falhas de toda a sua linhagem de dependências.

1.1 Como Corrigir

Se o Trivy apontar uma vulnerabilidade em uma biblioteca que não está declarada diretamente no seu arquivo de configuração, siga os passos abaixo para forçar a correção das dependências transitivas.

Correção no Backend (Java/Maven)

  1. Identifique na coluna "Fixed Version" a versão mínima que corrige a falha no relatório do Trivy.
  2. Acesse o MVN Repository e pesquise pelo nome da biblioteca (ex: org.apache.tomcat.embed:tomcat-embed-core)
    • Selecione a versão que o trivy sugeriu como "Fixed Version".
    • Confirme se o groupId e artifactId correspondem ao reportado pelo Trivy.
    • Copie o bloco de código da dependência XML (exemplo de bloco abaixo).
    3. Exemplo de Dependência
    <dependency>
    <groupId>org.apache.tomcat.embed</groupId>
    <artifactId>tomcat-embed-core</artifactId>
    <version>10.1.30</version>
</dependency>
  3. No seu pom.xml, localize a seção <dependencies> e cole o código copiado para forçar o override da versão transitiva.
  4. Na pasta do backend, execute o comando abaixo para aplicar as mudanças:
    5. Terminal
    mvn clean install -DskipTests

Correção de Vulnerabilidades (Frontend/Node.js)

Siga os passos abaixo para atualizar bibliotecas vulneráveis apontadas pelo Trivy.

  1. Identifique o nome da biblioteca e a Fixed Version no relatório do Trivy.
  2. Dependência Direta: Se a biblioteca estiver listada em "dependencies" no package.json, atualize a versão diretamente lá.
  3. Dependência Transitiva (Filha): Se a biblioteca não estiver lá, adicione o campo "overrides" entre "dependencies" e "devDependencies":
    4. package.json
    "dependencies": { ... },

"overrides": {
  "biblioteca-vulneravel": "^2.0.0"
},

"devDependencies": { ... }
  4. Aplique as mudanças: Exclua o arquivo package-lock.json e execute o comando abaixo para reinstalar e atualizar a árvore de dependências:
    5. Terminal
    npm install

2. Gerar Relatório (OPCIONAL)

GERAR RELATÓRIO EM CSV

Se você deseja gerar um relatório formatado para acompanhamento das vúlnerabilidades, siga os passos abaixo:

1. Scan e Output JSON 
trivy fs --scanners vuln --format json --output resultado_trivy.json .
2. Criação do Filtro JQ 
cat << 'EOF' > filtro.jq
["Biblioteca", "CVE-ID", "Severidade", "CVSS", "Vers. Instalada", "Vers. Corrigida", "Link"],
(.Results[]? | .Vulnerabilities[]? | [
    .PkgName, 
    .VulnerabilityID, 
    .Severity, 
    (.CVSS.nvd.V3Score // .CVSS.ghsa.V3Score // "--"), 
    .InstalledVersion, 
    .FixedVersion, 
    .PrimaryURL
]) | @csv
EOF
3. Conversão para CSV 
jq -r -f filtro.jq resultado_trivy.json > auditoria_libs.csv

3. O Arquivo .trivyignore

O arquivo .trivyignore serve para que o detetive ignore vulnerabilidades específicas que já foram analisadas e consideradas de baixo risco ou "falso positivo" pela equipe. As regras formais de exceção e rastreabilidade ficam centralizadas na Triagem de bibliotecas.

.trivyignore 
# CVE-2023-1234: Descrição do motivo
# Ignorar até que saia patch oficial (exp:2026-12-31)
CVE-2023-1234
CVE-2024-5678

* Toda entrada deve conter um comentário explicando o motivo da supressão.

4. Boas Práticas

CHECKLIST DO DETETIVE
  • Pre-commit: Rode o trivy antes de cada commit.
  • Integridade: Mantenha o .trivyignore sempre atualizado e documentado.
  • Decisão: O Trivy acha o erro, a Lógica de Biblioteca resolve o erro.

5. Tratamento de CVEs por Severidade

NÍVEL DO CVE AÇÃO IMEDIATA BIBLIOTECAS
CRITICAL CVE de Exploração Remota: Bloqueio obrigatório. Requer atualização imediata ou remoção da lib. Consulte a política de exceção na página de bibliotecas.
BLOQUEADO
HIGH CVE de Alto Impacto: Correção obrigatória no sprint atual. Exceção apenas conforme a política registrada na página de bibliotecas.
REVISAR
MEDIUM CVE Moderado: Correção recomendada. Exceção permitida se o esforço de migração for > 16h e o registro estiver na página de bibliotecas.
CONSULTAR
LOW CVE de Baixo Risco: Monitorar via deps.dev. Correção realizada apenas em janelas de manutenção.
PERMITIDO
O que é um CVE?

Common Vulnerabilities and Exposures (CVE) é uma lista de registros para vulnerabilidades de segurança cibernética publicamente conhecidas. Cada ID (ex: CVE-2024-1234) refere-se a uma falha específica em um software ou biblioteca.

6. Boas Práticas de Implementação

CHECKLIST TRIVY
  • Local: Executar trivy fs . antes de commits com novas dependências.
  • Pre-commit Hook: Bloquear commits com CRITICAL/HIGH vulnerabilities.
  • Pipeline: Gate obrigatório no CI/CD com --exit-code 1.
  • Imagens: Scan de todas as imagens antes de push para registry.
  • Base Images: Preferir imagens slim/alpine com menos vulnerabilidades.
  • DB Updates: Atualizar banco de vulnerabilidades diariamente.
  • SARIF Upload: Integrar com GitHub/GitLab Security Dashboards.
  • Exceções de bibliotecas: Documentar todas as exceções na página de bibliotecas com justificativa e rastreabilidade.
