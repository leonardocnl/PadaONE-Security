Trivy - Software Composition Analysis

O Trivy é a nossa principal ferramenta de visibilidade para a Supply Chain, identificando vulnerabilidades [cite: 3] em dependências, imagens de container e infraestrutura como código (IaC). [cite: 3]

TRIVY: O NOSSO DETETIVE

O Trivy atua como um investigador implacável: ele vasculha cada camada do projeto [cite: 3] para encontrar o crime (vulnerabilidade). No entanto, ele apenas reporta os fatos. [cite: 3] A sentença final — se vamos atualizar, ignorar ou mitigar — é ditada estritamente pela nossa [cite: 3] Lógica de Bibliotecas.

2. Instalação e Configuração

INSTALAÇÃO (WINDOWS/LINUX)

PowerShell / Bash 
# Windows (Chocolatey)
choco install trivy

# Linux (Debian/Ubuntu)
sudo apt-get install trivy

3. Comandos de Uso

ESCANEANDO DIRETÓRIOS E IMAGENS

Terminal 
# Diretório atual (Filesystem)
trivy fs ./

# Imagem Docker
trivy image nome-da-imagem:tag

FILTRANDO VULNERABILIDADES

# Apenas bibliotecas de linguagem
trivy fs ./ --scanners vuln --pkg-types library

# Por nível de severidade (Alta e Crítica)
trivy fs ./ --severity CRITICAL,HIGH

4. Avaliação dos Resultados após rodar o Trivy

Ao executar o scanner, o Trivy apresenta uma tabela de resultados parecida com este exemplo abaixo:

backend/package-lock.json (npm)
Total: 8 (UNKNOWN: 0, LOW: 0, MEDIUM: 4, HIGH: 4, CRITICAL: 0)
Library Vulnerability Severity Fixed Version
@remix-run/router CVE-2026-22029 HIGH 1.23.2
body-parser CVE-2025-13466 MEDIUM 2.2.1

Por que a biblioteca não está no meu arquivo?

É comum pesquisar por @remix-run/router no seu pom.xml ou [cite: 3] package.json e não encontrar resultados. [cite: 3] Isso ocorre porque você declara apenas as [cite: 3] bibliotecas "pai" (diretas).

[cite: 3] O Trivy, porém, reconstrói o grafo completo para identificar falhas em [cite: 3] dependências transitivas. [cite: 3] Ou seja, ele encontra vulnerabilidades nos "filhos" [cite: 3] e "netos" das bibliotecas que você instalou. [cite: 3]

Entenda a lógica: Quando o relatório aponta o seu arquivo de manifesto como o [cite: 3] Target, ele não está dizendo que o erro está escrito ali textualmente, mas sim que [cite: 3] aquele arquivo é a origem da cadeia de importação.

[cite: 3] Para o desenvolvedor, o desafio é identificar qual "pai" trouxe esse "filho" vulnerável. [cite: 3] Corrigir a versão da biblioteca pai no seu arquivo geralmente resolve as falhas de toda a sua linhagem de [cite: 3] dependências. [cite: 3]

4.1 Como Corrigir

Se o Trivy apontar uma vulnerabilidade em uma biblioteca que não está declarada [cite: 3] diretamente no seu arquivo de configuração, siga os passos abaixo para forçar a correção das [cite: 3] dependências transitivas. [cite: 3]

Correção no Backend (Java/Maven)

  1. Identifique na coluna "Fixed Version" a versão mínima que corrige a falha. [cite: 3]
  2. No seu pom.xml, localize a seção <properties>. [cite: 3]
  3. Adicione a tag específica da biblioteca com a versão sugerida pelo Trivy para forçar o [cite: 3] override. [cite: 3]
pom.xml
<properties>
    <!-- Exemplo: O Trivy acusou Tomcat 10.1.30 vulnerável e sugeriu a 11.0.12 -->
    <tomcat.version>11.0.12</tomcat.version>
</properties>

Correção no Frontend (Angular/Node.js)

  1. Identifique o nome da biblioteca vulnerável e a Fixed Version no relatório do [cite: 3] Trivy. [cite: 3]
  2. No seu package.json, adicione ou localize o campo "overrides". [cite: 3]
  3. Insira o nome da biblioteca "filha" e a versão corrigida conforme o exemplo abaixo. [cite: 3]
package.json
{
  "name": "daxa",
  "version": "1.6.0",
  "dependencies": { ... },
  "overrides": {
    "nome-da-lib-filha": "1.2.3" 
  }
}

Dica: Após salvar, execute npm install para aplicar as mudanças no [cite: 3] package-lock.json. [cite: 3]

5. Gerar Relatório (OPCIONAL)

GERAR RELATÓRIO EM CSV

Se você deseja gerar um relatório formatado para acompanhamento das vúlnerabilidades, siga os passos abaixo:

1. Scan e Output JSON 
trivy fs --scanners vuln --format json --output resultado_trivy.json .
2. Criação do Filtro JQ 
cat << 'EOF' > filtro.jq
["Biblioteca", "CVE-ID", "Severidade", "CVSS", "Vers. Instalada", "Vers. Corrigida", "Link"],
(.Results[]? | .Vulnerabilities[]? | [
    .PkgName, 
    .VulnerabilityID, 
    .Severity, 
    (.CVSS.nvd.V3Score // .CVSS.ghsa.V3Score // "--"), 
    .InstalledVersion, 
    .FixedVersion, 
    .PrimaryURL
]) | @csv
EOF
3. Conversão para CSV 
jq -r -f filtro.jq resultado_trivy.json > auditoria_libs.csv

6. O Arquivo .trivyignore

O arquivo .trivyignore serve para que o detetive ignore vulnerabilidades específicas que já [cite: 3] foram analisadas e consideradas de baixo risco ou "falso positivo" pela equipe. [cite: 3]

.trivyignore 
# CVE-2023-1234: Descrição do motivo
# Ignorar até que saia patch oficial (exp:2026-12-31)
CVE-2023-1234
CVE-2024-5678

* Toda entrada deve conter um comentário explicando o motivo da supressão. [cite: 3]

8. Tratamento de CVEs por Severidade

NÍVEL DO CVE AÇÃO IMEDIATA WAIVER
CRITICAL CVE de Exploração Remota: Bloqueio obrigatório. Requer atualização imediata ou [cite: 3] remoção da lib. [cite: 3]
PROIBIDO
HIGH CVE de Alto Impacto: Correção obrigatória no sprint atual. Aceita waiver apenas se [cite: 3] não houver patch. [cite: 3]
RESTRITO
MEDIUM CVE Moderado: Correção recomendada. Waiver permitido se o esforço de migração for > [cite: 3] 16h. [cite: 3]
PERMITIDO
LOW CVE de Baixo Risco: Monitorar via deps.dev. Correção realizada apenas em janelas de [cite: 3] manutenção. [cite: 3]
PERMITIDO
O que é um CVE?

Common Vulnerabilities and Exposures (CVE) é uma lista de registros para [cite: 3] vulnerabilidades de segurança cibernética publicamente conhecidas. [cite: 3] Cada ID (ex: CVE-2024-1234) [cite: 3] refere-se a uma falha específica em um software ou biblioteca. [cite: 3]

9. Tipos de Scan Avançados

SBOM GENERATION

Gerar Software Bill of Materials [cite: 3] 

trivy sbom --format cyclonedx \
  --output sbom.json \
  myapp:latest
IAC SCAN (TERRAFORM)

Scan de misconfigurations IaC [cite: 3] 

trivy config \
  --severity CRITICAL,HIGH \
  ./terraform/
KUBERNETES MANIFESTS

Scan de YAML do K8s [cite: 3]

trivy config \
  --scanners misconfig \
  k8s/deployment.yaml
ROOTFS SCAN

Scan de filesystem de container [cite: 3] 

trivy rootfs \
  --security-checks vuln \
  /path/to/rootfs

11. Boas Práticas de Implementação

CHECKLIST TRIVY
  • Local: Executar trivy fs . antes de commits com novas dependências. [cite: 3]
  • Pre-commit Hook: Bloquear commits com CRITICAL/HIGH vulnerabilities. [cite: 3]
  • Pipeline: Gate obrigatório no CI/CD com --exit-code 1. [cite: 3]
  • Imagens: Scan de todas as imagens antes de push para registry. [cite: 3]
  • Base Images: Preferir imagens slim/alpine com menos vulnerabilidades. [cite: 3]
  • DB Updates: Atualizar banco de vulnerabilidades diariamente. [cite: 3]
  • SARIF Upload: Integrar com GitHub/GitLab Security Dashboards. [cite: 3]
  • Waivers: Documentar todas as exceções no .trivyignore com [cite: 3] justificativa. [cite: 3]
